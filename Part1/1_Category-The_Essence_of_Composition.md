# Chapter 1: Category: The Essence of Composition — 합성, 카테고리의 본질

> **핵심 개념**: 카테고리는 대상(object)과 사상(morphism/arrow)으로 이루어진 구조다. 카테고리의 본질은 **합성(composition)**이며, 합성에는 **결합법칙**과 **항등 사상**이라는 두 가지 규칙이 있다. 프로그래밍에서 함수의 합성이 바로 이 구조에 해당한다.

---

## 카테고리란?

카테고리(Category)는 놀라울 정도로 단순한 개념이다. 다음 두 가지로 구성된다:

1. **대상(Object)**: 원자처럼 더 이상 내부를 들여다보지 않는 추상적 존재
2. **사상(Morphism)** 또는 **화살표(Arrow)**: 대상 사이의 관계를 나타냄. 모든 화살표에는 시작점(source/domain)과 끝점(target/codomain)이 있다

두 대상 사이에는 화살표가 없을 수도, 하나일 수도, 무한히 많을 수도 있다. 자기 자신으로 돌아오는 화살표도 가능하다.

그런데 카테고리의 진짜 핵심은 대상이나 화살표 자체가 아니라 **합성**이다. 대상 A에서 B로 가는 화살표 `f`와 B에서 C로 가는 화살표 `g`가 있으면, 반드시 A에서 C로 가는 화살표 `g ∘ f`가 존재해야 한다. 이것이 합성이다.

---

## 화살표와 함수

프로그래밍에서 화살표는 **함수**에 대응된다. 타입 A를 받아 타입 B를 반환하는 함수 `f`와, 타입 B를 받아 타입 C를 반환하는 함수 `g`가 있으면 이 둘을 합성할 수 있다.

수학에서 합성은 작은 원 기호 `∘`로 나타내며, `g ∘ f`는 "g after f"로 읽는다 (오른쪽에서 왼쪽 순서).

```haskell
f :: A -> B
g :: B -> C

-- 합성: g after f
g . f :: A -> C
```

```swift
func f(_ a: A) -> B { ... }
func g(_ b: B) -> C { ... }

// 합성
func gAfterF(_ a: A) -> C {
    return g(f(a))
}
```

---

## 합성의 두 가지 성질

### 1. 결합법칙 (Associativity)

세 개의 사상 `f`, `g`, `h`가 합성 가능할 때, 괄호를 어디에 묶든 결과는 같다:

```
h ∘ (g ∘ f) = (h ∘ g) ∘ f = h ∘ g ∘ f
```

```haskell
f :: A -> B
g :: B -> C
h :: C -> D

h . (g . f) == (h . g) . f == h . g . f
```

이 성질 덕분에 여러 함수를 합성할 때 괄호를 신경 쓸 필요가 없다. 결합법칙은 함수를 다룰 때는 매우 자명하지만, 다른 카테고리에서는 반드시 그렇지 않을 수 있다.

### 2. 항등 사상 (Identity Morphism)

모든 대상 A에는 자기 자신으로 돌아오는 특별한 화살표 **id_A**가 존재한다. 이 항등 사상은 합성의 단위 원소 역할을 한다:

```
f ∘ id_A = f
id_B ∘ f = f
```

프로그래밍에서 항등 함수는 입력을 그대로 반환하는 함수다:

```haskell
id :: a -> a
id x = x
```

```swift
func id<T>(_ x: T) -> T {
    return x
}
```

"아무것도 안 하는 함수가 왜 필요하지?"라고 생각할 수 있다. 숫자 0이나 1이 직접 계산 결과로는 안 쓰여도 수학 체계에서 없어서는 안 되는 것처럼, 항등 사상도 카테고리 구조를 완성하는 데 필수적이다.

## 합성과 프로그래밍

프로그래밍의 본질은 **분해(decomposition)**와 **합성(composition)**이다.

복잡한 문제를 마주하면 우리는 이를 작은 문제로 쪼갠다. 쪼갠 문제가 아직 크면 더 쪼갠다. 각 조각을 해결하는 코드를 작성한 뒤, 다시 합쳐서 전체 문제를 해결한다. 이 과정이 가능하려면 조각들이 **합성 가능**해야 한다. 분해할 수 없다면 합성도 불가능하고, 합성이 불가능한 방식으로 분해하는 것은 의미가 없다.

> 우리가 문제를 분해하고 다시 합성하는 과정은 컴퓨터를 위한 것이 아니다. 한 번에 처리할 수 있는 정보량이 제한된 인간의 두뇌를 위한 것이다.

적절한 합성 단위의 크기를 입체 물체에 비유하면:

- **표면적** = 합성을 위해 알아야 하는 정보 (인터페이스, 함수 시그니처)
- **부피** = 내부 구현을 위한 정보의 양

카테고리 이론은 대상의 내부를 들여다보는 것을 적극적으로 지양한다. 대상은 추상적이고 모호한 점일 뿐이다. **중요한 것은 대상들이 화살표를 통해 어떻게 연결되어 있는가**이다. 객체 지향에서는 클래스의 인터페이스가, 함수형에서는 함수의 선언(시그니처)이 이 "표면적"에 해당한다. 합성을 위해 내부 구현을 알아야 한다면, 추상화의 이점을 잃게 된다.

---

## 정리

- **카테고리**는 대상(object)과 사상(morphism)으로 구성된다
- 카테고리의 본질은 **합성(composition)**이다: `f :: A → B`와 `g :: B → C`가 있으면 `g ∘ f :: A → C`가 존재
- 합성은 **결합법칙**을 만족한다: `h ∘ (g ∘ f) = (h ∘ g) ∘ f`
- 모든 대상에는 **항등 사상** `id`가 존재한다: `f ∘ id = f`, `id ∘ f = f`
- 프로그래밍에서 대상은 **타입**, 사상은 **함수**에 대응된다
- 프로그래밍의 핵심은 문제를 분해하고 다시 합성하는 것이며, 카테고리 이론은 이 합성의 수학적 기반을 제공한다
