# Chapter 2: Types and Functions — 타입과 함수

> **핵심 개념**: 프로그래밍의 타입은 값들의 **집합(Set)**으로 볼 수 있고, 함수는 집합 사이의 사상이다. 타입 시스템은 합성 가능성을 보장하는 도구이며, 집합과 함수의 카테고리(**Set**)는 프로그래밍을 모델링하는 핵심 카테고리다.

---

## 왜 타입이 필요한가?

정적이고 강력한(static, strong) 타입 시스템은 프로그램의 정확성을 보장하는 **스펠 체커** 역할을 한다. 원숭이가 무작위로 타자를 쳐서 셰익스피어의 작품을 완성할 확률이 거의 없듯이, 타입 검사 없이 올바른 프로그램이 만들어질 가능성은 극히 낮다.

---

## 타입과 합성 가능성

카테고리 이론은 화살표(사상)의 합성에 관한 것이지만, 아무 두 화살표나 합성할 수는 없다. 합성하려면 첫 번째 화살표의 끝점(target)이 두 번째 화살표의 시작점(source)과 **같아야** 한다.

프로그래밍에서 이것은 곧: **한 함수의 반환 타입이 다음 함수의 인자 타입과 일치해야** 합성이 가능하다는 뜻이다.

- 강한 타입 시스템은 이 일치를 컴파일 타임에 검증해준다
- 타입 추론(type inference) 덕분에 프로그래머가 매번 타입을 명시하지 않아도 된다
- 유닛 테스트로 타입 검사를 대체할 수는 있지만, 컴파일러만큼 체계적이지는 못하다

---

## 타입이란 무엇인가?

프로그래밍에서 **타입은 값들의 집합**이다:

- `Bool` = `{True, False}` — 원소가 2개인 유한 집합
- `Char` = 유니코드 문자들의 집합
- `Integer` = 정수 집합 — 무한 집합

```haskell
x :: Integer  -- x는 정수 집합의 원소
```

이상적으로는 Haskell의 타입을 집합에, 함수를 집합 사이의 사상으로 대응시킬 수 있다. 하지만 수학적 함수는 코드를 실행하지 않는 반면, 프로그래밍의 함수는 계산을 수행해야 한다.

### 바닥 타입 (Bottom, ⊥)

무한 루프에 빠지거나 종료되지 않는 계산은 어떻게 모델링할까? 이를 위해 모든 타입에 **바닥(bottom, ⊥)**이라는 특별한 값을 추가한다. 바닥은 "끝나지 않는 계산"을 나타낸다.

```haskell
f :: Bool -> Bool
f x = undefined    -- undefined는 bottom으로 간주되며 타입 체크를 통과한다

f = undefined      -- Bool -> Bool 타입 자체가 undefined일 수도 있다
```

바닥을 포함하는 함수를 **부분 함수(partial function)**라 한다. 바닥이 없는 함수는 **전체 함수(total function)**다. 바닥의 존재 때문에 Haskell의 타입과 함수의 카테고리는 엄밀한 **Set** 대신 **Hask**로 불리지만, 실용적으로는 Set처럼 다뤄도 무방하다.

---

## Set의 카테고리

1장에서 카테고리는 "대상 + 화살표 + 합성 규칙"이라고 했다. 집합과 함수의 세계에 그대로 대입해보자:

| 카테고리 구성요소 | Set에서의 정체 |
|---|---|
| 대상 (점) | 각각의 집합 — `Bool = {True, False}`, `Int = {0, 1, -1, 2, ...}` 등 |
| 화살표 | 집합에서 집합으로 가는 함수 — `f: Int → Bool` 같은 것 |
| 합성 | 함수 합성 `g ∘ f` |
| 항등 화살표 | 항등 함수 `id x = x` |

결합법칙도 성립하고 항등 함수도 있으니까, 카테고리 조건을 전부 만족한다. 이 카테고리를 **Set**이라고 부른다. 프로그래밍에서 타입 = 집합, 함수 = 함수이므로, **우리가 코드에서 타입과 함수로 일하는 것은 Set 카테고리 안에서 일하는 것과 같다.**

카테고리적 관점에서는 집합의 내부 구조(원소)를 들여다보지 않는다. 각 집합은 하나의 점이 되고, 중요한 것은 집합들이 함수(화살표)로 어떻게 연결되는가이다.

### 왜 굳이 카테고리라는 틀에 넣는가?

카테고리 이론은 **"원소를 안 보고도 구조를 설명할 수 있는 언어"**다. 집합에서 증명한 성질이 카테고리 언어로 번역되면, 집합이 아닌 다른 세계에도 그대로 적용된다.

예를 들어 "곱(product)"이라는 개념을:
- Set에서 쓰면 → 튜플 `(A, B)`
- 타입에서 쓰면 → struct / pair
- 논리에서 쓰면 → AND

세 가지가 전혀 달라 보이지만, 카테고리 관점에서는 **같은 구조**다. 한 번 증명하면 세 곳에 다 적용된다.

Set은 그 출발점이다 — 가장 익숙한 세계(집합과 함수)에서 카테고리적 사고 방식을 연습하고, 나중에 그 사고방식을 다른 세계에 그대로 가져가는 것이다.

## 수학적 모델의 필요성

프로그램의 의미를 기술하는 방법으로 **표시 의미론(denotational semantics)**이 있다. 이는 모든 프로그래밍 구성 요소에 수학적 해석을 부여하여 프로그램을 증명 가능하게 만든다.

사이드 이펙트가 있는 프로그램은 어떻게 수학적으로 모델링할까? Eugenio Moggi가 발견한 것처럼, 계산 효과(computational effect)는 **모나드(Monad)**로 매핑될 수 있다. 이에 대해서는 나중에 자세히 다룬다.

---

## 순수 함수와 불순 함수

- **불순 함수(Dirty Function)**: 명령형 프로그래밍의 함수. 같은 입력에 다른 출력을 내거나 외부 상태를 변경할 수 있다. 수학적 함수가 아니다.
- **순수 함수(Pure Function)**: 동일 입력 → 동일 출력. 외부 상태에 영향을 주지 않는다. 수학적으로 모델링하기 쉽다.

Haskell에서는 모든 함수가 순수하다. 다른 언어에서는 순수한 부분 집합으로 제한하거나, 부수 효과를 별도로 다뤄야 한다.

## 타입의 예시

### Void — 공집합 (Empty Set)

원소가 없는 집합에 대응하는 타입. Haskell에서는 `Void`라 부른다 (C++의 `void`와는 다르다).

- `Void`를 인자로 받는 함수를 정의할 수는 있지만, 호출할 수는 없다 — `Void` 타입의 값을 만들 수 없기 때문
- 반환 타입은 무엇이든 될 수 있다 (다형적)

```haskell
absurd :: Void -> a   -- "불합리한" 전제에서는 무엇이든 따라 나온다
```

논리적으로 `Void`는 **거짓(False)**에 대응된다 (Curry-Howard 대응).

### Unit — 단원소 집합 (Singleton Set)

오직 하나의 값만 가지는 타입. Haskell에서는 `()`로 표기한다.

```haskell
() :: ()    -- 타입 ()의 유일한 값도 ()

-- Unit을 받아 정수를 반환: 순수함수라면 항상 같은 상수를 반환
fortyFour :: () -> Integer
fortyFour () = 44
```

- **Unit → a 함수**: 타입 a의 원소를 하나 골라내는(picking) 것에 해당. 이를 나중에 **대상의 일반화된 원소(generalized element)**라 부를 것이다
- **a → Unit 함수**: 모든 값을 `()`로 보내는 함수. 입력을 무시하므로 모든 타입에 대해 단 하나만 존재한다

```haskell
unit :: a -> ()
unit _ = ()     -- 모든 타입에 대해 parametrically polymorphic
```

논리적으로 `Unit`은 **참(True)**에 대응된다.

### Bool — 두 원소 집합

Haskell에서 Bool은 내장 타입이 아니라 대수적 데이터 타입으로 정의된다:

```haskell
data Bool = True | False
```

## 별첨: 함수, 전사, 단사

### 함수의 구조

집합 간의 함수에서:
- **정의역(Domain)**: 함수의 입력 집합
- **공역(Codomain)**: 함수의 출력이 속하는 집합
- **상(Image)**: 실제로 함수의 출력으로 나오는 값들의 집합 (공역의 부분집합)

핵심 성질: 정의역의 여러 원소가 공역의 하나의 원소로 매핑될 수 있지만, 하나의 원소가 여러 값으로 매핑될 수는 없다 (함수의 정의).

### 집합 관점: 단사, 전사, 동형사상

집합에서는 **원소 간의 대응 관계**로 성질을 정의한다.

**단사(Injective)**: 서로 다른 입력은 서로 다른 출력으로 간다. 공역의 각 원소에 대응되는 정의역 원소가 **최대 1개**. 정보가 뭉개지지 않으므로, 출력 값을 보면 어떤 입력에서 왔는지 확정할 수 있다. 단, 공역에 빈자리가 있을 수 있다.

**전사(Surjective)**: 공역의 모든 원소가 출력으로 나온다 (상 = 공역). 공역의 각 원소에 대응되는 정의역 원소가 **최소 1개**. 빈자리는 없지만, 여러 입력이 같은 출력으로 뭉개질 수 있다.

**동형사상(Isomorphism)**: 단사 + 전사. 공역의 각 원소에 대응되는 정의역 원소가 **정확히 1개**. 겹침도 빈자리도 없으므로 역함수가 존재한다.

```haskell
g . f = id_a    -- a에서 출발하여 a로 돌아옴
f . g = id_b    -- b에서 출발하여 b로 돌아옴
```

### 카테고리 관점: 단사사상(Mono), 전사사상(Epi), 동형사상(Iso)

카테고리에서는 원소를 들여다보지 않으므로, **다른 화살표와의 합성 관계**로 같은 성질을 정의한다.

**단사사상(Monomorphism)** — "f **앞에** 뭘 붙여도 구별된다":

> 모든 대상 c와 사상 `g1, g2 :: c → a`에 대해, `f ∘ g1 = f ∘ g2`이면 `g1 = g2`

f가 서로 다른 입력을 같은 출력으로 뭉개버린다면, g1과 g2가 다른 값을 보내도 f를 통과하면 구별이 안 된다. mono는 그런 뭉개기가 없다는 뜻이다 — 집합에서의 단사에 대응.

**전사사상(Epimorphism)** — "f **뒤에** 뭘 붙여도 구별된다":

> 모든 대상 c와 사상 `g1, g2 :: b → c`에 대해, `g1 ∘ f = g2 ∘ f`이면 `g1 = g2`

f가 공역의 일부만 채운다면, g1과 g2가 그 빈자리에서만 다르게 행동해도 f를 통과한 값들에 대해선 구별이 안 된다. epi는 그런 빈자리가 없다는 뜻이다 — 집합에서의 전사에 대응.

**동형사상(Isomorphism)** — "역방향 화살표가 존재한다":

> `g :: b → a`가 존재하여 `g ∘ f = id_a`이고 `f ∘ g = id_b`

mono + epi보다 강한 조건이다. 역화살표가 **실제로 존재**해야 한다.

### 집합 vs 카테고리 비교

| | 집합 관점 (원소를 봄) | 카테고리 관점 (화살표만 봄) |
|---|---|---|
| **단사 / Mono** | 다른 입력 → 다른 출력 | f 앞에 뭘 붙여도 구별됨 |
| **전사 / Epi** | 공역에 빈자리 없음 | f 뒤에 뭘 붙여도 구별됨 |
| **동형 / Iso** | 단사 + 전사 = 역함수 존재 | 역화살표가 존재 |

> **주의**: 집합의 카테고리(Set)에서는 Mono + Epi = Iso이지만, 일반 카테고리에서는 반드시 그렇지 않다! 카테고리에서의 동형은 반드시 역화살표의 존재를 직접 요구한다.

## 정리

- 프로그래밍의 **타입은 값들의 집합**이다
- **바닥(⊥)**은 끝나지 않는 계산을 나타내며, 모든 타입에 포함된다
- 타입 시스템은 함수 합성의 정합성을 컴파일 타임에 보장한다
- **Set 카테고리**: 대상=집합, 사상=함수. 프로그래밍의 기본 모델
- **Void**(공집합) = 논리적 거짓, **Unit**(단원소 집합) = 논리적 참
- **순수 함수**만이 수학적 함수에 대응된다. 사이드 이펙트는 모나드로 모델링 가능
- 동형사상 = 전단사(bijection) = 뒤집을 수 있는 사상
- **전사사상(Epi)**, **단사사상(Mono)**는 전사/단사의 카테고리적 일반화
